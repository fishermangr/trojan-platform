# AES-128 Side-Channel Analysis Platform

A complete platform for AES-128 ECB encryption on FPGA with power trace capture, built using:

- **Google XLS** — High-Level Synthesis (C++ → Verilog RTL)
- **Xilinx Vivado** — FPGA synthesis, implementation & bitstream generation
- **ChipWhisperer Husky** — Power trace capture & target communication
- **CW312-A35** (Artix-7 XC7A35T) on **CW313** carrier board — FPGA target

## Quick Start

### 1. Run HLS (C++ → Verilog)

```bash
cd AES-trojan/hls
python3 run_hls.py
```

This compiles `src/aes_xls.cc` through the XLS toolchain:
1. **xlscc**: C++ → XLS IR
2. **opt_main**: Optimize XLS IR
3. **codegen_main**: XLS IR → Verilog (`rtl/aes_128.v`)

Options:
```bash
python3 run_hls.py --xls-root /path/to/xls    # Custom XLS path
python3 run_hls.py --pipeline-stages 2          # 2-stage pipeline
python3 run_hls.py --dry-run                    # Print commands only
```

### 2. Synthesize & Generate Bitstream (Vivado)

```bash
cd AES-trojan
vivado -mode batch -source vivado/synthesize.tcl
```

This runs the full Vivado flow:
1. Reads CW reference HDL from `rtl/cw/` (with `SS2_WRAPPER` and `XLS_AES` defines)
2. Reads adapter + XLS core from `rtl/` (`xls_aes_adapter.v` + `aes_128.sv`)
3. Applies `cw312_a35.xdc` constraints for `ss2_aes_wrapper` top module
4. Synthesizes, places, routes
5. Generates bitstream → `build/ss2_aes_wrapper.bit`

Options:
```bash
vivado -mode batch -source vivado/synthesize.tcl -tclargs --rtl_dir /path/to/rtl
```

### 3. Upload Bitstream to FPGA

```bash
cd AES-trojan/scripts
python3 upload_bitstream.py --bitstream ../build/ss2_aes_wrapper.bit
```

This:
1. Connects to the Husky
2. Configures the HS2 clock generator (7.37 MHz)
3. Programs the CW312-A35 FPGA via SPI (CW312T_XC7A35T programmer)
4. Enables clock, resets FPGA, configures SS2 serial (TIO1=rx, TIO2=tx)
5. Reads identification registers (IDENTIFY, CRYPT_TYPE, CRYPT_REV)
6. Verifies AES-128 against NIST test vectors

### 4. Run Encryption & Capture Power Traces

```bash
cd AES-trojan/python

# Single encryption with NIST test vectors
python3 cw_aes.py encrypt

# Custom key and plaintext
python3 cw_aes.py encrypt --key 00112233445566778899aabbccddeeff \
                           --data 00000000000000000000000000000000

# Capture 100 traces with random plaintexts
python3 cw_aes.py encrypt --random --num-traces 100

# Full configuration
python3 cw_aes.py encrypt --random --num-traces 500 \
                           --num-samples 5000 \
                           --sample-rate 29480000 \
                           --gain-db 25 \
                           --output-mat ../data/traces.mat \
                           --output-json ../data/traces.json
```

## Detailed Documentation

### AES-128 XLS Source (`src/`)

The TinyAES C implementation has been rewritten for Google XLS `xlscc` compatibility:

- **No pointers** — all operations use fixed-size arrays
- **No preprocessor conditionals** — AES-128 ECB only
- **XLS channel I/O** — `__xls_channel` for key, plaintext, ciphertext
- **Loop unrolling** — all loops annotated with `#pragma hls_unroll yes`
- **Class-based design** — `AesEcb128Encrypt` class with `#pragma hls_top`

The AES implementation is functionally identical to the original TinyAES:
- S-box lookup via `sbox_lookup()` function with static const array
- Standard AES-128 key expansion (KeyExpansion)
- Standard 10-round Cipher: SubBytes → ShiftRows → MixColumns → AddRoundKey

### RTL Architecture (`rtl/`)

```
                    CW312-A35 FPGA (XC7A35T)
  ┌─────────────────────────────────────────────────────┐
  │  ss2_aes_wrapper (top)                              │
  │  ┌───────┐   ┌──────────────────────────────────┐   │
  │  │ SS2   │   │  cw305_top                        │   │
  │  │ UART  │──→│  ┌────────────┐  ┌─────────────┐ │   │
  │  │ COBS  │   │  │ usb_reg_fe │  │ reg_aes     │ │   │
  │  │ CRC   │←──│  │ (addr dec) │──│ (key/pt/ct) │ │   │
  │  └───────┘   │  └────────────┘  └──────┬──────┘ │   │
  │  rxd (IO2)   │                         │        │   │
  │  txd (IO1)   │  ┌──────────────────────┴──────┐ │   │
  │              │  │ xls_aes_adapter             │ │   │
  │              │  │ (start/done ↔ rdy/vld)      │ │   │
  │              │  │  ┌────────────────────────┐  │ │   │
  │              │  │  │ aes_128 (XLS-generated)│  │ │   │
  │              │  │  └────────────────────────┘  │ │   │
  │              │  └──────────────────────────────┘ │   │
  │              └──────────────────────────┬────────┘   │
  │                                        │ TRIGGER    │
  │  CLK ← HS2 (7.37 MHz from Husky)      └──→ IO4     │
  └─────────────────────────────────────────────────────┘
```

- **`ss2_aes_wrapper.v`**: Top-level connecting SS2 protocol to CW305 register infrastructure
- **`ss2.v`**: SimpleSerial2 protocol handler (UART @ 230400 baud + COBS + CRC framing)
- **`cw305_top.v`**: Register front-end, clock management (IBUFG/BUFG/BUFGMUX)
- **`cw305_reg_aes.v`**: AES register file with key, plaintext, ciphertext, go/busy registers
- **`xls_aes_adapter.v`**: Bridges CW305 start/done interface to XLS ready/valid channels
- **`aes_128.sv`**: (Generated by XLS) AES-128 core with ready/valid channel I/O

### Communication Protocol (SS2 + CW305 Registers)

The FPGA uses ChipWhisperer's SimpleSerial2 (SS2) protocol over UART at 230400 baud with COBS encoding and CRC. The Python `CW305` target class provides `fpga_write()`/`fpga_read()` for register access.

| Register | Address | Size | Description |
|----------|---------|------|-------------|
| CLKSETTINGS | 0x00 | 1 | Clock mux configuration |
| USER_LED | 0x01 | 1 | LED control |
| CRYPT_TYPE | 0x02 | 1 | Crypto type (2 = AES) |
| CRYPT_REV | 0x03 | 1 | Crypto revision (5) |
| IDENTIFY | 0x04 | 1 | ID register (0x2E) |
| CRYPT_GO | 0x05 | 1 | Write: start encryption; Read: busy status |
| CRYPT_TEXTIN | 0x06 | 16 | 128-bit plaintext input |
| CRYPT_CIPHEROUT | 0x09 | 16 | 128-bit ciphertext output |
| CRYPT_KEY | 0x0A | 16 | 128-bit key input |

### Trigger Signal

The trigger output (IO4) goes **active high** when the AES encryption begins and returns low when the ciphertext is ready. This signal tells the Husky exactly when to sample power consumption.

### Pin Mapping (CW312-A35 on CW313)

FPGA Part: **XC7A35T-1CSG324** (324-pin BGA)

| Signal | FPGA Pin | CW313 Pin | Description |
|--------|----------|-----------|-------------|
| `clk` | D15 | HS2/CLKIN | Clock from Husky (7.37 MHz) |
| `clkout` | A13 | — | Clock output (optional) |
| `resetn` | A16 | nRST | Active-low reset (driven by Husky) |
| `rxd` | V11 | IO2 | SS2 UART: Husky TX → FPGA RX |
| `txd` | V10 | IO1 | SS2 UART: FPGA TX → Husky RX |
| `io3` | V12 | IO3 | Unused (active low pulldown) |
| `io4` | V14 | IO4 | Trigger: active high during encrypt |
| `led1` | R1 | — | USB clock heartbeat |
| `led2` | V2 | — | SS2 error indicator |
| `led3` | V5 | — | Crypto clock heartbeat |

**CRITICAL**: Scope TIO must be swapped: `scope.io.tio1='serial_rx'`, `scope.io.tio2='serial_tx'`

### Power Trace Capture

The `cw_aes.py` script configures the Husky for power analysis:

| Parameter | CLI Flag | Default | Description |
|-----------|----------|---------|-------------|
| Clock freq | `--clk-freq` | 7370000 | Target clock in Hz |
| Sample rate | `--sample-rate` | auto | ADC sample rate in Hz |
| ADC multiplier | `--adc-mul` | 4 | ADC clock = clkgen × mul |
| Samples/trace | `--num-samples` | 5000 | ADC samples per capture |
| Pre-trigger | `--pre-samples` | 0 | Samples before trigger |
| Gain | `--gain-db` | 25.0 | ADC gain in dB |
| Timeout | `--timeout` | 2000 | Trigger timeout in ms |

### Output Files

All data is saved in the `data/` directory:

- **`.mat` file** (MATLAB/SciPy format): Contains numpy arrays for:
  - `key` — 16-byte key
  - `plaintexts` — (N × 16) uint8 array
  - `ciphertexts_hw` — (N × 16) uint8 array (from FPGA)
  - `ciphertexts_sw` — (N × 16) uint8 array (from PyCryptodome)
  - `traces` — (N × num_samples) float64 array (power traces)
  - Metadata: sample rate, gain, timestamps, etc.

- **`.json` file**: Human-readable record of all operations:
  - Key, plaintexts, ciphertexts (as hex strings)
  - Per-trace HW vs SW match status
  - Configuration parameters

### Software Verification

Every hardware encryption result is compared against PyCryptodome's AES-128 ECB:
- Mismatches are counted and reported
- The JSON file records per-trace match/mismatch status
- NIST SP 800-38A test vectors are used for initial verification

## Troubleshooting

### HLS fails with xlscc errors
- Ensure XLS is built: `cd xls && bazel build //xls/contrib/xlscc //xls/tools:opt_main //xls/tools:codegen_main`
- Check include paths: `xls_int.h` should be at `xls/xls/contrib/xlscc/synth_only/xls_int.h`

### Vivado synthesis fails
- Ensure `aes_128.v` exists in `rtl/` (run HLS first)
- Check that pin assignments match your CW312-A35 board revision
- Verify `CFGBVS` and `CONFIG_VOLTAGE` settings

### No trigger detected
- Verify IO4 is connected (CW313 → Husky)
- Check `scope.trigger.triggers = "tio4"` in the capture script
- Use an oscilloscope to probe the trigger pin

### SS2 communication issues
- **TIO swap**: Must set `scope.io.tio1='serial_rx'` and `scope.io.tio2='serial_tx'` (FPGA txd is on IO1)
- **Clock order**: Configure `clkgen_freq` before programming, disable `hs2` during SPI, re-enable after
- **Reset**: Drive `scope.io.nrst='low'` then `'high'` after clock is enabled
- **Baud rate**: SS2 uses 230400 baud (pBIT_RATE=32 @ 7.37 MHz); mismatch causes timeouts
- Use `scripts/test_ss2.py` for debugging communication issues

### Ciphertext mismatch
- Verify the AES XLS source matches TinyAES byte ordering
- Check that the key is loaded before sending plaintext
- Run with `--num-traces 1` and compare against known test vectors
